Memory Capabilities in Agentic Frameworks and Standalone Systems

Overview of Memory in Agentic Frameworks

Modern agentic AI frameworks integrate various memory mechanisms to help language-model-based agents retain information and improve over time. Key memory types include short-term (working) memory for immediate context, long-term memory for persistent knowledge, semantic memory for facts and concepts, episodic memory for past events or interactions, and procedural memory for learned skills or rules. Below we compare how four popular frameworks – Anthropic Claude’s Agent SDK, LangChain’s LangGraph (with LangMem), CrewAI, and Google’s Agent Development Kit (ADK) – handle these memory types. We also examine standalone memory systems (Mem0, Letta, Zep, LangMem SDK) that can augment or integrate with such frameworks. Finally, we clarify how “skills” (the tools or actions an agent can perform) differ from “memory” (the information an agent retains).

Claude Agent SDK (Anthropic)

Memory Handling: By default, Claude agents have no built-in long-term memory beyond the current conversation window. The Claude Agent SDK addresses this with a Memory Tool that lets the agent store and retrieve data to a file directory on the client side. Short-term memory is simply the conversation context (the recent messages and the agent’s chain-of-thought, up to Claude’s context window limit). Long-term memory is achieved by persisting information across sessions: the memory tool allows Claude to create/read/update files in a /memories directory, giving it a knowledge base it can consult beyond a single session. This means developers can implement semantic memory (facts or notes) by writing them to memory files, or episodic memory by logging events/outcomes for future reference. For example, Anthropic’s engineers have the agent write a claude-progress.txt log after each coding session – a persistent artifact that the next session’s agent reads to understand what has been done. This log serves as episodic memory of past actions. Procedural memory (rules for how to do tasks) in Claude’s case is largely contained in the prompt (the system instructions) or the model’s weights; the SDK doesn’t automatically update the agent’s “skills” on its own, but developers can manually refine prompts over time.

Self-Improvement & Evolution: Claude’s framework supports agent evolution through memory in a rudimentary but effective way: the agent can learn from past interactions by reading what it wrote to the memory files. For instance, it can improve consistency on a long project by recalling previously discovered facts or previously written code. Anthropic has also introduced context compaction and multi-session strategies to help Claude work on long tasks without forgetting earlier steps. In practice, developers orchestrate self-improvement by feeding Claude its own outputs from prior sessions (via the memory files or progress logs) so it avoids repeating mistakes and builds on earlier work. However, the Claude Agent SDK does not yet implement automatic prompt optimization or skill learning loops – any major improvement in the agent’s core behavior usually comes from the developer updating the system prompt or providing feedback between runs. The memory tool’s main role is to ensure continuity and learning across sessions, e.g. remembering user preferences, past plans, or intermediate results so Claude performs better over time.

LangChain LangGraph (with LangMem)

Memory Handling: LangChain’s new LangGraph framework provides a structured approach to agent memory. It cleanly separates short-term memory (called thread-scoped memory) and long-term memory. Short-term memory is the ongoing conversation state within a single session (a “thread”). LangGraph automatically checkpoints this state (message history, any interim data) to a database, allowing the agent to resume a conversation or process without losing context. Long-term memory in LangGraph is namespace-scoped, meaning information can persist across sessions or conversations and be queried when needed. LangChain explicitly models human-like memory types in its LangMem SDK:

Semantic memory (facts) – storing factual information or extracted knowledge about the world or user. For instance, the agent can remember a user’s preferences or key domain facts not contained in the base model. LangMem provides a MemoryManager to extract and save such facts during conversations. This often uses a vector store or database of facts the agent accumulates.

Episodic memory (experiences) – retaining records of specific past interactions or events. In LangChain, episodic memory might take the form of saving important examples or transcripts from previous episodes that the agent can refer to as needed. (LangMem currently doesn’t have a heavy built-in module for episodic memory beyond letting you store conversation data; the design encourages distilling episodes into either semantic facts or few-shot examples for later use.)

Procedural memory (how-to/skills) – internalized knowledge of how to perform tasks or rules the agent should follow. In LangChain’s view, an agent’s procedural knowledge lives partly in its system prompt or policy and can be optimized over time. The LangMem SDK actually includes a Prompt Optimizer that can update the agent’s instructions based on past successes/failures. This is effectively procedural memory management: the agent learns better “skills” or rules (e.g. “if the user asks about astronomy, always mention moons if relevant”) by analyzing interaction trajectories and adjusting its core prompt. Procedural memory in LangChain is thus not a data store but a mechanism to evolve the agent’s behavior.


LangChain’s memory system allows hot-path updates (the agent decides to remember something during an interaction) as well as background updates (asynchronous processing to update memories after an interaction). All memories can be persisted in stores (vector databases, JSON, etc.) and recalled via LangChain’s retrieval functions. The framework even draws analogies to human memory – for example mapping semantic to factual knowledge, episodic to “things I did”, and procedural to ingrained instructions. In summary, short-term memory is handled via conversation state management, while long-term memory is handled via the LangMem SDK and vector stores, categorized into semantic/episodic/procedural for different uses.

Self-Improvement & Evolution: LangChain explicitly supports agent self-improvement through memory. The procedural memory approach is a form of self-evolution – using feedback loops to refine the agent’s prompt/policy over time. For example, LangMem can take a set of past dialogues and outcomes (success or failure scores) and produce an improved system prompt that biases the agent toward more effective behavior. This means a LangChain agent can learn from its mistakes: if it consistently misses certain details, the Prompt Optimizer can add a rule to cover that gap in the future. Semantic memory also aids improvement by accumulating facts the agent didn’t originally “know,” thereby expanding its effective knowledge without retraining the model. Although LangChain provides the tools, the developer must decide when to invoke these learning processes (e.g. periodically run a prompt optimization, or store a new fact). Episodic memory (storing and recalling specific examples) can improve performance via better few-shot context – the agent can refer back to how it solved a similar problem before, guiding it to repeat or avoid past actions. Overall, LangChain’s memory framework is designed for lifelong learning: an agent that remembers user-specific info and past solutions to become more personalized and competent over time.

CrewAI

Memory Handling: CrewAI’s framework includes a built-in memory system that covers short-term, long-term, and entity memory out of the box. By enabling memory=True on an agent, developers get default implementations of these:

Short-Term Memory: CrewAI uses a Retrieval-Augmented Generation (RAG) approach for short-term context. Recent interactions and results are embedded into a ChromaDB vector store, allowing the agent to retrieve relevant recent content when generating a response. This helps maintain coherence within the current conversation or task execution by recalling pertinent details even if the raw history is large (similar to a message buffer with semantic search).

Long-Term Memory: CrewAI preserves cross-session knowledge in a SQLite database. Notably, it stores task results and insights from past runs so that an agent can build up knowledge over time. For example, if an agent accomplished a multi-step job or found a solution in a previous session, the outcome can be logged to long-term memory (SQLite) and later retrieved. This is analogous to episodic memory – the agent can recall what it learned or concluded before in order to inform future tasks. The use of a structured database means this memory can store text, summaries, or any data the developer chooses to log.

Entity Memory: CrewAI also offers a specialized memory for entities (people, places, concepts), using RAG with a vector store to keep track of information about specific entities encountered. This functions like a mini knowledge graph: as the agent encounters a new person or concept, it can store facts or descriptions about them, enabling deeper understanding and consistency when that entity comes up again. In essence, this is semantic memory targeted at entities – facts and relationships are embedded so the agent can later retrieve “who is Alice?” or “what is Project X?” from its memory.


CrewAI combines these into a Contextual Memory module that pulls from short-term, long-term, and entity stores as needed to maintain contextual continuity. By default, CrewAI stores the memory data in local files (using OS-specific app data directories) – for example, it creates folders for knowledge/, short_term_memory/, long_term_memory/, and entities/ with the relevant ChromaDB or SQLite files. This transparency allows developers to inspect or persist the memories easily.

Importantly, CrewAI also supports external memory providers. Developers with specialized needs can disable the basic memory and plug in an external memory service via an ExternalMemory interface. For instance, CrewAI provides integration with Mem0 (see below) – in a few lines, one can connect Mem0’s memory model as the agent’s memory backend. External memory might be used to get more advanced semantic search, larger-scale vector databases, or cloud-based memory sharing across applications. A comparison in the documentation shows that while the built-in memory is simple and local, using an external memory can enable user-specific personalization and custom storage at the cost of more setup. In summary, CrewAI covers short-term and long-term memory needs by default and gives the flexibility to extend or replace the memory subsystem with external tools (vector DBs, memory services) as needed.

Self-Improvement & Evolution: CrewAI’s memory system contributes to agent improvement primarily by accumulating knowledge. Each successful task outcome or user interaction can be logged in long-term memory, so the agent doesn’t repeat work or mistakes. For example, if an agent already solved a sub-problem in a previous run, it can recall that solution instead of recalculating it, thus improving efficiency and competence over time. This is a form of learning from experience. Because CrewAI’s short-term memory uses semantic search, the agent can also avoid losing important context in long conversations – it “forgets” less and can bring relevant past points back when needed. That improves coherence and prevents the agent from contradicting itself or asking the same questions again.

CrewAI does not (as of writing) have an explicit module for autonomous prompt evolution or skill learning (no direct counterpart to LangChain’s prompt optimizer). Self-improvement is therefore realized implicitly: the agent’s knowledge grows with each interaction. Notably, if integrating an external memory like Mem0, the agent can benefit from pattern learning (e.g. Mem0 can infer user habits – more on this below – which CrewAI’s agent could use to adjust its suggestions). In that sense, pairing CrewAI with a system like Mem0 could give a personalization feedback loop, where the agent adapts to the user’s behavior over time. Overall, CrewAI’s design leans toward reliability and accumulation – making sure valuable information is retained so the agent incrementally improves its answers and avoids past pitfalls. Any more advanced self-evolution (like rewriting its own goals or adding new skills) would currently have to be implemented by the developer on top of the provided memory mechanisms.

Google Agent Development Kit (ADK)

Memory Handling: Google’s ADK (often used with Vertex AI) approaches agent memory via the concepts of session state and memory services. A session in ADK represents one contiguous interaction (e.g. a chat session) and includes a history of events (messages) and a modifiable state (key–value memory within that conversation). This is essentially short-term memory: the agent’s immediate context for the current conversation. ADK’s session state can track things like the user’s name or progress through a task within that session. By default, this short-term memory lives in-memory (RAM) and is lost when the session ends or the server restarts. However, ADK provides a SessionService that can be backed by a database to persist session data if needed (for production deployments). ADK even has “magic” prefixes for state keys (like user:, app:) that allow certain pieces of state to persist across sessions, e.g. a user’s setting that should apply every time. This bridges short-term and long-term a bit by letting some session data become semi-permanent.

For true long-term memory, ADK offers the MemoryService abstraction. There are two implementations:

In-Memory MemoryService: Simply stores all past session transcripts in memory for search. This is mainly for prototyping – it retains full conversation logs (raw chat history) and only supports basic keyword search for retrieval. It’s akin to a naive episodic memory: the agent can query past chats by keyword, but there’s no intelligent processing of those logs.

Vertex AI Memory Bank Service: This is a more advanced, cloud-managed memory. The Memory Bank uses a large model (Google’s Gemini model) to extract meaningful information from conversations and store those distilled “memories” in a persistent store. In other words, instead of keeping full transcripts, it picks out the key facts or outcomes (a semantic summary) and accumulates those. The agent can later query this memory by using a semantic search interface – the Memory Bank supports vector similarity search over the stored memories. For example, if across chats the user mentioned their favorite color or a specific problem, the Memory Bank would have a record of that fact which can be retrieved even in a new session. This corresponds to semantic memory (facts about the user or context) gleaned from episodic interactions. It’s also a form of long-term episodic memory, since it keeps track of what happened “many conversations ago,” but in a summarized way.


Using the MemoryService, an ADK agent can automatically save a session to long-term memory (usually at the end of the session) and later load relevant info when a new session starts or when needed. In practice, developers set up a callback or configuration so that once a conversation is done, the agent calls add_session_to_memory() to ingest that session’s info into the Memory Bank. When the agent needs to answer a question in a new session, it can call a memory tool (like load_memory) or a search function to retrieve any matching facts from the Memory Bank. The retrieved snippets are then added to the prompt so the agent “recalls” them. This design cleanly separates short-term vs. long-term: short-term is the session (transient but can be DB-backed), and long-term is the Memory Bank (persistent and summarized).

Memory Types: While ADK’s documentation doesn’t explicitly label memory as semantic/episodic/procedural, we can map them. Semantic memory is represented by the distilled facts in Memory Bank (it stores general knowledge the agent learned about the user or context, not entire episodes). Episodic memory is present in that the agent can refer to specific past events, but typically via the semantic extraction (for example, “what was the user’s score last session?” would be stored as a fact). If needed, one could use raw transcripts (episodic) via the in-memory service, but that’s less scalable. Procedural memory (rules for task execution) is not really handled by ADK automatically – the agent’s “skills” or behaviors are defined by its prompt and code. ADK is more about orchestration, so it doesn’t have a built-in mechanism for an agent to learn new procedures on its own. Developers would have to update the agent’s prompt or give it new tools (skills) if they want to change its procedural knowledge.

Self-Improvement & Evolution: Google’s ADK enables continuous improvement mainly through better context and personalization. By using the Memory Bank, an agent can learn from past conversations – e.g. remembering a user’s preferences, past issues, or progress – which makes future interactions more personalized and efficient. For instance, an agent could gradually build a profile of a student’s learning difficulties and adapt its tutoring strategy accordingly in subsequent sessions. This is a form of self-improvement, albeit guided by the developer’s setup of what to store. The agent is not rewriting its own code or prompt, but it is improving its answers by having access to a growing knowledge base of prior interactions. The Memory Bank’s use of AI to summarize means the agent’s long-term memory stays concise and focused, avoiding overwhelming the model with too much irrelevant old context. In terms of agent evolution, ADK’s philosophy is to give the agent the right information rather than change the agent’s inherent reasoning abilities. There is no built-in analog of a prompt optimizer or reflective loop in ADK as of now. However, a developer could manually analyze stored conversations and fine-tune the model or adjust prompts externally. To summarize, ADK’s agent evolution through memory manifests as better contextual awareness and personalization over time (the agent “remembers you”), rather than the agent autonomously learning entirely new behaviors.
